# リポジトリ構造の説明

「Shuntaのクリエイティブラボ日記」リポジトリの **ディレクトリ設計の意図** を残しておくドキュメント。
"なぜこの構造にしたか" を未来の自分に伝えるためのもの。

## 全体像

```
.
├── README.md                  ← 入り口
├── LICENSE
│
├── lore/                      ← 設定資料（世界 / キャラ / 物語）
├── design-system/             ← ビジュアル統一のためのルール
├── episodes/                  ← 各話（1 話 = 1 フォルダ = 1 画像）
├── ideas/                     ← ネタ帳・バックログ
├── assets/                    ← キャラ立ち絵・コンセプトボード
├── meta/                      ← 制作論・運用ドキュメント
└── .github/                   ← Issue / PR テンプレート
```

## 階層分けの考え方

トップレベルは "**役割で割る**" 方針:

| ディレクトリ      | 担っている役割                                       |
| ----------------- | ---------------------------------------------------- |
| `lore/`           | "世界の正" を置く（変えたら必ず PR）                  |
| `design-system/`  | "見た目の正" を置く                                   |
| `episodes/`       | "アウトプットそのもの"                                |
| `ideas/`          | "まだアウトプットになっていないもの"                  |
| `assets/`         | "繰り返し使う画像素材"（キャラ立ち絵 / コンセプト）    |
| `meta/`           | "リポジトリ自体の運用ルール"                          |
| `.github/`        | GitHub の機構（Issue / PR テンプレ）                  |

> "設定" と "アウトプット" を **同じ階層に並べない** ことで、
> 「どこを編集しているか」の精神的なモード切替を作っている。

## lore/ の中の構造

```
lore/
├── README.md           ← lore の入り口・コアコンセプト
├── world.md            ← 世界観・舞台
├── origin.md           ← 起源・参画ストーリー
├── timeline.md         ← 年表（相対表記）
├── lexicon.md          ← 用語集・表記ルール
├── relationships.md    ← 人間関係マップ
├── characters/         ← キャラ別プロフィール
└── storyline/
    ├── README.md       ← 大方針
    ├── seasons/        ← S1〜S4 の詳細プロット
    └── arcs/           ← 縦軸の長期伏線（bugmaru など）
```

- **横軸（シーズン）と縦軸（アーク）を別ディレクトリ** に分ける
  → "今やってる S2 のテーマ" と "ずっと走るバグまる伏線" を別管理できる
- **キャラと関係性は別ファイル** にする
  → 1 キャラのファイルが膨れすぎず、関係性の俯瞰も独立して読める
- **lexicon.md** で表記ゆれを一箇所に集約

## episodes/ の構造

```
episodes/
├── README.md
├── _template/             ← 新規エピソード用テンプレ
│   └── episode.md
└── NNNN-slug/             ← 1 話 = 1 フォルダ
    ├── episode.md         ← 脚本・コマ割り・メモ
    ├── cover.png          ← 完成版 1 枚画像（SNS 投稿用）
    └── work/              ← ネーム / 下書き（任意）
```

### 設計意図

- **1 話 = 1 フォルダ** にすることで、脚本と画像が co-locate される
  - 旧構造（`assets/episodes/NNNN/sheet-*.png` が `episodes/NNNN-*.md` と離れる）の不便を解消
- **完成版は必ず `cover.png`** に統一
  - フォルダ名から完成画像のパスが一意に決まる
  - SNS 投稿スクリプトを書くときに楽
- **`work/` は任意**
  - 公開する必要がないネームを置く場所
  - リポジトリに残すか `.gitignore` するかは運用で決められる

### "リリース単位" を導入しない理由

過去には「テーマリリース（記事 + 関連話 N 本）」という上位単位を持っていたが、
記事との対応をやめて **1 話 = 1 リリース** に統一したため、
`releases/` のような上位ディレクトリは持たない。
連作したい場合は front matter `continues_from` で繋ぐ。

## assets/ の構造

```
assets/
├── README.md
├── characters/            ← キャラ別の立ち絵・表情差分
│   ├── shunta/ / iris/ / mika/ / takuma/ / bugmaru/
└── concept/               ← コンセプト・ムードボード
```

- **エピソードごとの画像はここではなく `episodes/NNNN-slug/` に置く**
- ここに置くのは **「繰り返し参照する素材」** のみ（キャラ立ち絵 / コンセプトアート）

## meta/ の役割

- リポジトリの "**運用ルール**" を置く
- ワークフロー、構造説明、運用指針
- "**作品** ではなく **リポジトリ** に関するドキュメント" の置き場

## なぜ `docs/` をやめたか

- 旧 `docs/` には世界観もワークフローも混ざっていた
- "設定（lore）" と "運用（meta）" は読み手が違う ので分けた
  - lore: 物語の整合性が気になる人が読む
  - meta: リポジトリ運用が気になる人が読む

## 命名規則

- **ディレクトリ名・ファイル名**: 小文字 + ハイフン（kebab-case）
- **エピソードフォルダ**: `NNNN-slug`（4 桁通し番号 + 内容を表す英語スラッグ）
- **完成画像**: `cover.png` で統一
- **下書き**: `work/sketch-*.png`
- **front matter の id**: 数字 4 桁文字列（`"0001"` 形式を推奨）

## 構造を変えたいとき

- 必ず PR にする
- このファイルも一緒に更新する（"変えた理由" を含めて）
- 過去エピソードへのパス影響を確認する
- Claude Code に整合性チェックを依頼する
